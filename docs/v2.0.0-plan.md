# xk6-kafka v2.0.0 - Detailed Implementation Plan

## 1. Goals

- Replace `segmentio/kafka-go` runtime paths with `confluent-kafka-go/v2` (`librdkafka`) while preserving existing JS users through compatibility shims.
- Standardize config parsing, harden runtime behavior, and close known lifecycle/error consistency gaps.
- Keep migration predictable with explicit deprecation windows, metric compatibility mapping, and CI/release reproducibility.

## 2. Dependency and Versioning Strategy

```go
// go.mod
remove github.com/segmentio/kafka-go
remove github.com/segmentio/kafka-go/sasl/aws_msk_iam_v2
add github.com/confluent/confluent-kafka-go/v2
remove github.com/riferrei/srclient
```

- If the Go API is breaking for importers, adopt `/v2` module path semantics.
- Pin and document supported `librdkafka` versions (Linux/macOS/Windows notes).
- Explicitly document CGO requirements in build docs and CI.
- Use `confluent-kafka-go/v2/schemaregistry` as the primary Schema Registry client in v2.

## 4. Core Interfaces (Target)

```go
type Producer interface {
    Produce(ctx context.Context, msgs []Message) error
    Flush(ctx context.Context) error
    Close() error
    Stats() ProducerStats
}

type Consumer interface {
    Consume(ctx context.Context, limit int) ([]Message, error)
    Seek(partition int, offset int64) error
    Position(partition int) (int64, error)
    CommitOffsets(ctx context.Context) error
    Close() error
    Stats() ConsumerStats
}

type AdminClient interface {
    CreateTopic(ctx context.Context, config TopicConfig) error
    DeleteTopic(ctx context.Context, name string) error
    ListTopics(ctx context.Context) ([]TopicInfo, error)
    GetMetadata(ctx context.Context, topic string) (*TopicMetadata, error)
    Close() error
}
```

Notes:

- Transactions are planned as optional/experimental for v2.0 and can move to v2.1 if they block core migration.
- `Consume(ctx, limit)` requires internal poll/drain strategy and deterministic shutdown behavior.
- Async API decision: keep existing synchronous APIs for backward compatibility, and add Promise-based async methods as opt-in for producer/admin/schema operations.

### 4.1 Async/Promise API Decision

- v2 will be hybrid:
  - keep sync APIs (`produce`, `consume`, admin operations) for compatibility and deterministic behavior
  - add async Promise APIs as additive methods (`produceAsync`, `createTopicAsync`, SR async operations)

- Aligns with modern xk6 extension ergonomics without forcing immediate migration.
- Preserves existing script behavior and timing assumptions.
- Enables better handling of non-blocking network-heavy operations.

Guardrails:

- Async paths must implement bounded concurrency and backpressure controls.
- `flush()` and `close()` semantics must explicitly drain/resolve pending async operations.
- Error mapping remains consistent with `Xk6KafkaError` typed failures.
- Async implementation must use one per-VU task queue/worker model (no cross-VU shared queue) to keep deterministic ownership.
- Promise cancellation must be wired to `context.Context` and k6 VU cancellation; unresolved promises at shutdown must fail fast with typed timeout/cancel errors.
- Add an async conformance test matrix: success, broker timeout, cancellation, close-with-pending, and backpressure saturation.

## 5. Configuration Refactor and Mapping

### 5.1 Standardize Parsing

- Replace mixed decode styles with one consistent decode path for writer/reader/admin configs.
- Validate unknown/invalid fields early and return typed errors (no nil pointer panics).

### 5.2 Unified Config Management Strategy

Avoid an ever-growing list of strongly-typed struct fields while still providing safe defaults and validation.

- Use a hybrid config model for each client type (producer/consumer/admin):
  - `Known` typed options: curated high-value fields we document and validate (for example brokers, topic, auth, tls, batch/delivery semantics).
  - `Raw` passthrough options: `map[string]any` for advanced Confluent/Kafka properties not yet promoted to typed fields.
- Merge precedence:
  1. hard safety defaults from xk6-kafka
  2. typed `Known` options
  3. user-provided `Raw` passthrough overrides
- `Raw` can override all keys.
- Add allowlist + denylist validation for `Raw`:
  - allow known Confluent key patterns (for example `acks`, `compression.type`, `fetch.*`, `socket.*`, `sasl.*`, `ssl.*`)
  - deny dangerous/unsupported overrides that break runtime assumptions
- Keep client-specific config wrappers:
  - `ProducerConfig` -> `kafka.ConfigMap`
  - `ConsumerConfig` -> `kafka.ConfigMap`
  - `AdminConfig` -> `kafka.ConfigMap`
  - all through one shared converter utility to avoid duplicate mapping logic

Promotion policy:

- New Kafka/Confluent options start in `Raw`.
- Promote to `Known` only when one of these is true:
  - required for most users
  - required for safety/correctness
  - repeatedly requested in issues/docs/examples
- Maintain a generated or curated mapping table in docs for:
  - typed options
  - passthrough examples
  - deprecated aliases

Validation and observability:

- Emit warnings for unknown passthrough keys (config typo detection), but allow opt-in strict mode to fail fast.
- Record effective config snapshots in debug logs with sensitive fields redacted.
- Keep redaction rules for secrets (`sasl.password`, tokens, private key material).
- Since `Raw` can override all keys, enforce post-merge runtime validation gates (required keys present, forbidden combinations rejected, known unsafe values blocked with typed errors).

### 5.3 Confluent Mapping Appendix (Required in code/docs)

Example mapping baseline:

```go
// Producer mapping examples
"bootstrap.servers" -> strings.Join(cfg.Brokers, ",")
"acks"              -> "0" | "1" | "all"
"batch.size"        -> cfg.BatchBytes
"linger.ms"         -> cfg.LingerMs
"compression.type"  -> cfg.Compression
```

- Include idempotence/retry-related keys and consumer group keys in the final mapping table.
- Document removed/changed behavior such as JS custom balancer callback if not supported on Confluent path.

## 6. Runtime Hardening (Must Do Before Broad Refactor)

1. Guard constructor/config parsing in writer/reader/topic constructors.
2. Fix admin controller connection lifecycle so temporary discovery connections are closed safely.
3. Replace panic paths with `Xk6KafkaError` typed failures.
4. Scope schema registry client/cache so multiple instances do not conflict through module-global mutable state.
5. Keep init-context protections and strengthen teardown/close guidance.

## 7. Schema Registry and Protobuf Strategy

- Protobuf constant is currently exported but serde is not fully implemented.
- Replace `srclient` usage with Confluent Schema Registry client (`confluent-kafka-go/v2/schemaregistry`) and Confluent serdes-compatible flow.
- Introduce an internal `SchemaRegistryClient` abstraction so implementation can be swapped without changing JS-facing APIs.
- Protobuf is planned and does not block v2.0 GA.
- v2.0 scope: ship Avro/JSON on Confluent SR path and keep clear Protobuf "planned" status.
- v2.1 target: implement dynamic descriptor-based Protobuf support and message-index handling.
- Document Confluent wire format handling, including message indexes behavior if Protobuf is supported.

## 8. Metrics Design and Compatibility

- Introduce Confluent-backed stats extraction path.
- Keep existing metric names where practical for dashboard compatibility.
- Publish a compatibility table:
  - unchanged metric names
  - renamed metrics
  - removed metrics with alternatives
- Ensure stats registration returns concrete errors (no silent unwrap-nil behavior).

### 9.1 E2E and Integration Testing Strategy

Test pyramid for v2:

1. **Unit tests (fast, deterministic)**
   - Serializer/deserializer behavior, config merge/validation, error mapping.
   - No network or container dependency.

2. **Integration tests (module + broker behavior)**
   - JS API behavior via `scripts/v2/*.js`.
   - Cover producer/consumer/admin flows, auth/tls variants, schema registry happy paths.
   - Keep these runnable locally and in CI as non-flaky smoke + medium suites.

3. **E2E tests (real infrastructure with Testcontainers)**
   - Provision Kafka + Schema Registry (and required dependencies) in isolated testcontainers network.
   - Validate end-to-end workflows: topic lifecycle, produce/consume, schema register/lookup/compatibility, teardown.
   - Validate failure behavior: broker restarts, unavailable registry, auth misconfig, timeout/cancel handling.

Test suites and ownership:

- `scripts/v2/smoke/*.js`: fast CI smoke coverage for core API shape.
- `scripts/v2/integration/*.js`: richer behavior tests for config/auth/schema permutations.
- `internal/test/e2e/*_test.go`: testcontainers-driven end-to-end assertions and environment orchestration.

Reliability controls:

- Use deterministic topic naming per test run to avoid cross-test interference.
- Enforce explicit setup/teardown and close/flush checks.
- Add retry policies only where broker metadata propagation is expected; no blanket retries.
- Track flaky tests separately and block release tags on unresolved flakes.

### 9.2 Versioned API Docs Strategy (`api-docs/v2`)

Documentation layout:

- Keep v1 docs intact and publish v2 docs under `api-docs/v2/`.
- Store v2 declaration/types and generated references in versioned paths to avoid accidental overwrite of v1 artifacts.
- Ensure migration docs link to both versions:
  - v1 reference for legacy users
  - v2 reference for new APIs and async additions

Generation flow:

- Add docs generation commands that target v2 output paths explicitly.
- Keep a stable entrypoint for latest docs while preserving versioned snapshots.
- Fail CI if API docs generation changes are detected but not committed.

Compatibility requirements:

- v2 API docs must include deprecated v1 aliases and their migration targets.
- `scripts/v2` examples should reference v2 docs paths and symbols.
- Release notes should include direct links to `api-docs/v2` for each v2 tag.

### 11.1 Schema Registry API Migration (`srclient` -> Confluent)

| Existing intent (`srclient`) | Confluent SR target | Notes for v2 implementation |
| --- | --- | --- |
| create SR client from URL and auth | `schemaregistry.NewConfig(url)` + `schemaregistry.NewClient(cfg)` | centralize client creation in `schema_registry.go`; avoid module-global mutable client state |
| get latest schema by subject | `client.GetLatestSchemaMetadata(subject)` | normalize into internal `SchemaMetadata` struct so JS API remains stable |
| get schema by id | `GetSchema(id)` | maintain local cache keyed by `(subject,id)` and global-id fallback |
| create/register schema under subject | `client.Register(subject, schemaInfo, normalize)` | preserve existing subject naming behavior (`TopicNameStrategy`, etc.) |
| schema compatibility checks | `client.TestCompatibility(subject, schemaInfo)` | map failures to `Xk6KafkaError` codes, not generic errors |
| resolve schema references | Confluent schema references model in schema info | preserve recursive reference resolution with cycle guards and bounded cache |
| serialize payload with schema id wire header | Confluent-compatible wire format handling | keep magic-byte + schema-id behavior consistent across Avro/JSON/Protobuf |
| protobuf message indexes support | Confluent protobuf serializer/index metadata | planned for v2.1; document placeholders and non-support behavior in v2.0 |

Implementation guardrails:

- Keep an internal `SchemaRegistryClient` interface in `interfaces.go` to decouple JS API from provider implementation.
- Do not expose Confluent SDK types directly through JS-facing structs.
- Add parity tests that replay current `schema_registry_*_test.go` behavior against Confluent-backed client.
- Add migration notes for auth headers, TLS, and subject-lookup differences.

## 15. Repo Strategy for Hosting v1 and v2

Use a single repository with one active default branch and one maintenance branch.

- Use `main` for v2 development and release tags.
- Create `release/v1` (or `maintenance/v1`) for v1 critical fixes only.
- Tag v1 releases from `release/v1` (for example `v1.9.x`) and v2 releases from `main` (for example `v2.0.x`).

Go module versioning requirements:

- Keep v1 module path as-is on v1 maintenance branch.
- Use semantic import versioning for v2:
  - `module github.com/mostafa/xk6-kafka/v2`
- Ensure all docs/examples for v2 use `/v2` import/build references.

Release and support policy:

- Publish a short support policy in `README.md` and `MIGRATION.md`:
  - v1: maintenance-only (security and critical bug fixes)
  - v2: active development
- Backport only high-priority fixes from `main` to `release/v1` where applicable.

### 15.1 Local Development Workflow for v2.0.0

Use this workflow for each implementation phase in this plan:

1. Start from local `v2.0.0`.
2. Create a phase branch from `v2.0.0` (for example `v2.0.0-phase-1-foundation`).
3. Implement only that phase scope and commit on the phase branch.
4. Squash-merge the phase branch into local `v2.0.0`.
5. Repeat for the next phase by creating a new branch from the updated local `v2.0.0`.

Rules:

- Do not push branches or tags to GitHub during phase execution.
- Keep all work local until all phases are complete and validated.
- After final validation, perform one coordinated push/publish step.

# `k6 x kafka validate` Schema Validation CLI

## Summary

- Add a new k6 subcommand extension registered as `kafka`, so users can run `k6 x kafka validate`.
- First release ships one command only: `validate`. It replaces the old standalone helper use case by checking that candidate payloads and schemas won't crash the xk6-kafka serde path.
- Scope is Avro and JSON Schema only. Protobuf stays out of scope and remains tracked in `docs/v2.1.0-protobuf.md`.
- The command must support both local schema files and Schema Registry lookups.

## Implementation Changes

- Register a subcommand extension with `go.k6.io/k6/subcommand.RegisterExtension("kafka", ...)`.
- Expose a parent command `k6 x kafka` and add one child command:
  - `k6 x kafka validate`

- `validate` command contract:
  - Required: `--schema-type avro|json`
  - Required: `--data <path|->`
    - `-` means read one payload from stdin
    - payload format is a single JSON value, not NDJSON or arrays of records
  - Exactly one schema source mode must be chosen:
    - Local mode:
      - `--schema-file <path>`
      - Optional repeated `--schema-dir <path>` for Avro dependency discovery only
    - Registry mode:
      - `--registry-url <url>`
      - `--subject <name>`
      - Optional `--version <latest|int>`, default `latest`
      - Optional registry auth/TLS flags mirroring current config shape:
        - `--registry-username`
        - `--registry-password`
        - `--tls`
        - `--insecure-skip-tls-verify`
        - `--tls-min-version`
        - `--client-cert-pem`
        - `--client-key-pem`
        - `--server-ca-pem`

- Validation behavior:
  - Avro:
    - Compile and resolve the schema first.
    - Validate by running the actual Avro serialize path with `hamba/avro` against the candidate JSON payload.
    - Success means the payload serializes successfully with the resolved schema.
  - JSON Schema:
    - Compile and resolve the schema first.
    - Validate by running the current JSON Schema validation path with `jsonschema`.
    - Success means the payload validates and can be serialized as JSON by the existing serde path.
  - No Kafka broker interaction is involved.
  - No schema creation or registration is performed in Schema Registry mode; registry mode is read-only.

- Loader design:
  - Add a runtime-independent validation service instead of tying the CLI to `sobek` or VU state.
  - Reuse existing `Schema`, `SchemaRegistryClient`, `Schema.Codec()`, `Schema.JSONSchema()`, and serde implementations where possible.
  - Add pure helpers for:
    - loading a local Avro root schema plus optional dependent `.avsc` files
    - compiling a local JSON Schema root file with filesystem `$ref` resolution
    - fetching registry schemas by subject/version into the existing `Schema` model without JS runtime dependencies
  - Keep the current JS API unchanged; this feature is CLI-only.

- Command output:
  - Default output is human-readable text only.
  - Success prints a short pass message with schema source, schema type, and serialized payload size.
  - Failure prints one summary line plus the underlying validation or compile error.
  - Exit code is non-zero on any schema resolution, compile, parse, or validation failure.
  - Do not add a machine-readable JSON output mode in the first release.

- Repo and docs updates:
  - Add CLI command implementation and tests.
  - Update `README.md` with a new "Schema validation CLI" section and examples for:
    - local nested Avro
    - local JSON Schema
    - Schema Registry subject validation
  - Update docs to replace references to the old external helper for basic validation workflows with the new built-in command.

## Test Plan

- Unit tests:
  - flag validation and mutually exclusive source selection
  - local Avro success and failure with nested referenced `.avsc` files
  - local JSON Schema success and failure with filesystem `$ref`
  - registry Avro success and failure using mock or `httptest` registry client paths
  - registry JSON Schema success and failure using mock or `httptest` registry client paths
  - TLS and basic auth flag mapping into `SchemaRegistryConfig`
  - stdin payload handling

- Command integration tests:
  - construct the `kafka` subcommand tree and verify `validate` is mounted under `k6 x kafka`
  - execute `validate` with sample files and assert stdout, stderr, and exit behavior
  - verify invalid schema or data returns non-zero without panicking

- Acceptance scenarios:
  - nested Avro payload that used to require `nested-avro-schema` validates successfully
  - malformed Avro payload fails with field-level context
  - invalid JSON payload against JSON Schema fails with path and context
  - registry subject lookup by latest and explicit version both work
  - no JS imports or Kafka brokers are required to use the command

## Assumptions and Defaults

- First release is intentionally one command, not multiple commands.
- `lint` and `test` are deferred; the parent `k6 x kafka` command should be structured so they can be added later without reshaping the UX.
- Local payload input is one JSON value only.
- Local Avro dependency search uses `--schema-dir`; JSON Schema local refs use normal filesystem-relative resolution from the root schema file.
- Schema Registry mode supports current basic auth and PEM-based TLS options, but not JKS-specific CLI flags in the first release.
- No public JS or TypeScript API changes are part of this feature.

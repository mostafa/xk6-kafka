# xk6-kafka v2.1.0 - Protobuf Design Plan

## Summary

- Add full `SCHEMA_TYPE_PROTOBUF` support in `v2.1.0`.
- Support Protobuf on both Schema Registry and standalone `SchemaRegistry` flows.
- Keep the existing JS entrypoints centered on `SchemaRegistry.serialize()` and `SchemaRegistry.deserialize()`.
- Use Confluent-compatible framing and Schema Registry behavior without requiring generated Go protobuf types in user projects.

## Public API

### Existing API kept

- `SCHEMA_TYPE_PROTOBUF` remains the public schema type selector.
- `SchemaRegistry.createSchema()` and `SchemaRegistry.getSchema()` remain the Schema Registry entrypoints.
- `SchemaRegistry.serialize()` and `SchemaRegistry.deserialize()` remain the encode and decode entrypoints.

### Additions to JS-facing schema metadata

Extend the existing schema payload with the following fields:

- `messageName?: string`
  - Fully qualified protobuf message name.
  - Required when a `.proto` schema defines more than one valid top-level message and the target cannot be inferred.
- `dependencies?: Record<string, string>`
  - Map of import path to raw `.proto` source.
  - Used only for standalone non-Schema-Registry protobuf resolution.

### Additions to JS-facing container metadata

Extend the existing serialize and deserialize container payload with:

- `protobufFormat?: "object" | "bytes"`
  - Default: `"object"`.
  - `"object"` encodes and decodes plain JS objects using protojson semantics.
  - `"bytes"` accepts or returns raw protobuf binary payloads.

## Behavior

### Default protobuf object behavior

- JS object mode follows canonical protojson behavior.
- Output uses protojson JSON field naming.
- Output enums are strings.
- Output bytes fields are base64 strings.
- Output `oneof` values contain only the selected field.
- Output omits unset fields by default.
- Well-known types use standard protojson forms.
- Unknown fields fail fast with typed errors.

### Standalone protobuf

- Standalone protobuf is supported through `new SchemaRegistry()` without a configured URL.
- Standalone protobuf accepts inline `.proto` schema text through `schema.schema`.
- Multi-file standalone protobuf uses `schema.dependencies`.
- Standalone protobuf does not perform Schema Registry registration or lookup.
- Standalone framing stays aligned with the existing non-Schema-Registry schema flow.
- Default standalone schema ID is `0` unless explicitly supplied in schema metadata.

### Schema Registry protobuf

- Schema Registry protobuf uses the existing `SchemaRegistryClient` abstraction.
- Subject naming strategies remain:
  - `TopicNameStrategy`
  - `RecordNameStrategy`
  - `TopicRecordNameStrategy`
- For protobuf, record-based subject naming uses `messageName`.
- If `messageName` is omitted and more than one candidate message exists, encoding fails with a typed configuration error.

### Bytes mode

- `serialize(..., protobufFormat: "bytes")` accepts:
  - `[]byte`
  - `Uint8Array`
  - byte arrays
  - base64 strings
- Raw binary input is validated against the resolved protobuf descriptor before framing.
- `deserialize(..., protobufFormat: "bytes")` returns the unframed protobuf payload bytes after schema framing is removed.

## Implementation

### Serdes runtime

- Add `ProtobufSerde` to the internal serdes registry.
- Remove the current protobuf planned-error path from active encode and decode flows.
- Keep Avro and JSON behavior unchanged.

### Descriptor runtime

Extend schema runtime state to cache:

- compiled file descriptors
- resolved message descriptors
- dynamic message factories
- dependency and import resolution artifacts

Use:

- `google.golang.org/protobuf/types/dynamicpb`
- descriptor-based runtime resolution
- Confluent protobuf serde and descriptor utilities where they fit cleanly

### Schema Registry integration

- Keep `SchemaRegistryClient` as the provider abstraction.
- Register protobuf schemas with `schemaType=PROTOBUF`.
- Preserve per-instance cache scoping.
- Reuse Schema Registry references for protobuf imports.
- Keep Confluent-compatible schema ID framing and protobuf message-index handling.

### Standalone integration

- Compile the root `.proto` schema and inline `dependencies` with no filesystem dependency.
- Resolve imports entirely from in-memory schema text.
- Support standalone encode and decode for both object mode and bytes mode.

### Errors

Add protobuf-specific typed failures for:

- schema compile failure
- missing import or dependency
- ambiguous or missing `messageName`
- invalid message-index path
- object-to-protobuf validation failure
- unsupported input type for the selected `protobufFormat`

## Internal Rules

### Object mode serialization

- Resolve the descriptor and target message.
- Convert the JS object into a dynamic protobuf message using protojson semantics.
- On Schema Registry path:
  - register or resolve the schema ID as needed
  - emit Confluent protobuf framing with schema ID and message indexes
- On standalone path:
  - emit local framed payload with schema ID `0` by default
  - still include protobuf message indexes

### Bytes mode serialization

- Accept raw protobuf bytes or equivalent binary input.
- Validate by decoding against the resolved descriptor.
- Reframe using the same Schema Registry or standalone path rules as object mode.

### Object mode deserialization

- Resolve the descriptor from Schema Registry metadata or standalone schema text.
- Remove framing and protobuf message indexes.
- Decode into a dynamic protobuf message.
- Return a protojson-shaped JS object.

### Bytes mode deserialization

- Resolve schema and strip framing.
- Return the raw protobuf payload bytes without object conversion.

## Tests

### Unit coverage

- descriptor compilation from inline `.proto`
- multi-file import resolution from `dependencies`
- Schema Registry reference resolution
- `messageName` inference and ambiguity failures
- object-mode round trips for:
  - enums
  - maps
  - repeated fields
  - nested messages
  - `oneof`
  - bytes
  - well-known types
- bytes-mode round trips
- subject naming with protobuf
- standalone framing with schema ID `0`
- Schema Registry framing with schema ID and message indexes

### Integration coverage

Add versioned scripts for:

- Schema Registry protobuf round trip
- standalone protobuf round trip
- imported schema or reference flows
- protobuf keys and values on Kafka produce and consume flows

### Acceptance criteria

- `SCHEMA_TYPE_PROTOBUF` works through the existing `SchemaRegistry` JS API in both Schema Registry and standalone modes.
- No generated Go protobuf code is required from users.
- Wire framing is compatible with Confluent protobuf framing, including message indexes.
- Documentation and API docs fully describe protobuf object mode and bytes mode.
- Existing protobuf planned-status docs and tests are replaced by real support coverage.

## Assumptions

- File name: `docs/v2.1.0-protobuf.md`
- Target release: `v2.1.0`
- First protobuf release includes full dynamic support rather than a reduced MVP.
- Standalone protobuf uses inline schema text and inline dependency maps instead of filesystem paths or descriptor-set blobs.
- The public API remains centered on existing `SchemaRegistry` methods rather than adding a separate protobuf client.
